Classer exposes an OpenAI-compatible generation surface so existing tools and
SDKs work unchanged. Point any client at `https://api.classer.ai/v1` with your
Classer API key and call `chat/completions`, `responses`, or `models` exactly
as you would against OpenAI.

<Note>
**Drop-in replacement for OpenAI clients.** Set `base_url` to
`https://api.classer.ai/v1` and `api_key` to your Classer key — no other code
changes needed.
</Note>

## Available models

| Model | Input ($/1M tokens) | Output ($/1M tokens) | Best for |
|-------|---------------------|----------------------|----------|
| **gpt-5.4** | $1.62 | $6.50 | Complex reasoning, long-form responses, agents |

[`GET /v1/models`](#list-models) to enumerate the live deployments.

## Chat completions

`POST /v1/chat/completions` accepts the [OpenAI Chat Completions request
shape](https://platform.openai.com/docs/api-reference/chat/create) verbatim
and returns the upstream response unchanged. The fields most relevant to
Classer are documented below; everything else is passed through.

<ParamField body="model" type="string" required>
  One of the [available models](#available-models). Unknown values return a
  `400` with the allow-list.
</ParamField>

<ParamField body="messages" type="array" required>
  Standard OpenAI message array (`role` + `content`).
</ParamField>

<ParamField body="max_completion_tokens" type="integer">
  Maximum output tokens. For reasoning models this also caps internal
  reasoning tokens, so set generously or `content` may come back empty.
</ParamField>

<ParamField body="stream" type="boolean" default="false">
  When `true`, returns Server-Sent Events. Classer auto-injects
  `stream_options.include_usage=true` so the final SSE chunk always carries
  the `usage` field for billing transparency. The flag is forced even if the
  client supplies `false`.
</ParamField>

<ParamField body="stream_options" type="object">
  OpenAI's `stream_options` is forwarded to the upstream. The
  `include_usage` field is always set to `true` regardless of input.
</ParamField>

### Response

For non-streaming requests, the response body is the full OpenAI Chat
Completions object:

```json
{
  "id": "chatcmpl-...",
  "model": "gpt-5.4-2026-03-05",
  "choices": [{
    "message": { "role": "assistant", "content": "..." },
    "finish_reason": "stop"
  }],
  "usage": {
    "prompt_tokens": 11,
    "completion_tokens": 5,
    "total_tokens": 16
  }
}
```

Billing is computed from `usage.prompt_tokens` and `usage.completion_tokens`
(applying the per-model rates above) and deducted from your balance within
~30 seconds of the request completing. See [Pricing](/pricing) for details.

### Reasoning models

Reasoning models — like `gpt-5.5` (when enabled) — silently generate hidden
"reasoning" tokens before any visible output. Azure rolls these into
`completion_tokens`, so a 200-token reply may have only ~100 visible tokens
and 100 reasoning tokens. **You are billed for the full `completion_tokens`
count.** Two practical implications:

1. Set `max_completion_tokens` generously, or visible content will be empty
   while internal reasoning consumes the budget.
2. Time-to-first-token (TTFT) is higher because reasoning happens before the
   first content delta streams.

### Examples

#### Basic chat completion

<CodeGroup>

```python Python (OpenAI SDK)
from openai import OpenAI

client = OpenAI(
    base_url="https://api.classer.ai/v1",
    api_key="$CLASSER_API_KEY",
)

resp = client.chat.completions.create(
    model="gpt-5.4-mini",
    messages=[{"role": "user", "content": "Summarise this in one sentence: ..."}],
    max_completion_tokens=200,
)
print(resp.choices[0].message.content)
print(resp.usage)
```

```bash cURL
curl -X POST https://api.classer.ai/v1/chat/completions \
  -H "Authorization: Bearer $CLASSER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-5.4-mini",
    "messages": [{"role": "user", "content": "Summarise this in one sentence: ..."}],
    "max_completion_tokens": 200
  }'
```

</CodeGroup>

#### Streaming

<CodeGroup>

```python Python (OpenAI SDK)
from openai import OpenAI

client = OpenAI(
    base_url="https://api.classer.ai/v1",
    api_key="$CLASSER_API_KEY",
)

stream = client.chat.completions.create(
    model="gpt-5.4",
    messages=[{"role": "user", "content": "Write a haiku."}],
    stream=True,
)
for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)
    if chunk.usage:
        print(f"\n[billed: {chunk.usage.prompt_tokens} in / {chunk.usage.completion_tokens} out]")
```

```bash cURL
curl -N -X POST https://api.classer.ai/v1/chat/completions \
  -H "Authorization: Bearer $CLASSER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-5.4",
    "messages": [{"role": "user", "content": "Write a haiku."}],
    "stream": true
  }'
```

</CodeGroup>

The final SSE event carries `usage`:

```
data: {"choices":[{"delta":{"content":"..."}}]}
data: {"choices":[{"delta":{"content":"..."}}]}
data: {"choices":[],"usage":{"prompt_tokens":11,"completion_tokens":17,"total_tokens":28}}
data: [DONE]
```

## Responses

`POST /v1/responses` is the OpenAI [Responses API](https://platform.openai.com/docs/api-reference/responses).
Same models, same pricing as chat completions; the request and response shapes
match OpenAI's spec. Streaming via `response.completed` events emits `usage`
natively, no injection needed.

```bash cURL
curl -X POST https://api.classer.ai/v1/responses \
  -H "Authorization: Bearer $CLASSER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-5.4-mini",
    "input": "Translate '\''hello'\'' to French."
  }'
```

## List models

`GET /v1/models` returns the live deployment list. Use this to discover which
generative models your account can call.

<CodeGroup>

```bash cURL
curl https://api.classer.ai/v1/models \
  -H "Authorization: Bearer $CLASSER_API_KEY"
```

```python Python (OpenAI SDK)
from openai import OpenAI

client = OpenAI(base_url="https://api.classer.ai/v1", api_key="$CLASSER_API_KEY")
for m in client.models.list().data:
    print(m.id)
```

</CodeGroup>

<ResponseExample>
```json
{
  "object": "list",
  "data": [
    {"id": "gpt-5.4",      "object": "model", "owned_by": "azure"},
    {"id": "gpt-5.4-mini", "object": "model", "owned_by": "azure"},
    {"id": "stimy-vl",     "object": "model", "owned_by": "self-hosted"}
  ]
}
```
</ResponseExample>

## Errors

| Status | Code | Cause |
|--------|------|-------|
| 401 | `INVALID_API_KEY` | Missing or invalid `Authorization: Bearer …` header |
| 400 | `invalid_request_error` | `model` not in the allow-list (response includes the list of valid models) |
| 400 | `invalid_request_error` | `model` field missing entirely |
| 402 | `INSUFFICIENT_BALANCE` | Account balance is below the grace buffer; top up to resume |
| 503 | – | Generate gateway not configured / unreachable |

## Rate limits

Generative endpoints share the per-API-key rate limits documented in
[Pricing](/pricing#rate-limits). Token usage from generative requests counts
against the same per-minute token bucket as classification.
